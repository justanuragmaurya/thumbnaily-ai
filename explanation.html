Diff → system shape

From inline generation to Redis, queues, and a worker

The refactor splits request handling from heavy work: Postgres stays the source of truth, Redis backs caches and BullMQ jobs, and a dedicated worker runs prompt enhancement, FAL, R2 uploads, and final writes. No Kafka — Redis + Postgres + explicit tables carry durability and idempotency.

01 What changed at a glance

Before

  • /api/generate-thumbnail authenticated, enhanced prompts, called FAL, uploaded to R2, wrote DB, and deducted credits inside a fire-and-forget async IIFE.
  • Progress lived in an in-memory Map (lost on restart, not shared across instances).
  • Public reads and credits hit Postgres repeatedly; client appCache was still per-tab memory.
  • Dodo webhook called payment logic synchronously without durable idempotency keys.

After

  • The route creates a GenerationJob, reserves one credit atomically, writes initial Redis progress, and enqueues BullMQ; the worker does AI + storage + DB completion.
  • Progress is stored in Redis (TTL) with Postgres fallback for polling.
  • Hot reads use Redis cache-aside (explore first page, detail, my thumbnails, homepage links, credits).
  • Webhooks upsert WebhookEvent and enqueue payment processing; worker grants credits and marks processed.

02 Architecture & flows

Target topology

Browser ──► Next.js (auth, UI, light APIs) │ ┌───────────┼───────────┬──────────────────┐ ▼ ▼ ▼ ▼ Redis Postgres BullMQ queues R2 (presigned uploads from web) cache / │ │ job progress │ └──► Worker (prompt → FAL → R2 → DB) Dodo webhook ──► Next verifies ──► WebhookEvent + enqueue ──► Worker credits user

Thumbnail generation sequence

  1. POST /api/generate-thumbnail: session check → validate prompt / reference URLs → transaction: findUnique user → conditional updateMany (credit ≥ 1) decrement → create GenerationJob (queued).
  2. setGenerationProgress in Redis; invalidate credits cache for that email; getThumbnailGenerationQueue().add(...) with stable job id. If enqueue fails, job is marked failed and credit refunded.
  3. Worker consumes job: updates job status, enhances prompt (OpenRouter), runs FAL, downloads image, uploads via presigned R2 URL, creates Thumbnails + reference rows in a transaction, marks job complete, updates Redis progress, busts explore/home/my-thumbnails caches.
  4. On final failure after retries: job → failed, credit refunded, Redis progress shows error.
  5. GET same route with progressId: read Redis first; else load job row from Postgres for resilient polling.

Billing sequence

  1. Verify Standard Webhooks signature; on payment success, WebhookEvent upsert by provider + providerEventId.
  2. If status is pending, enqueue payment job (idempotent job id).
  3. Worker increments credits from cart line items and marks event processed; clears credits Redis key for email.

Other fixes bundled in the diff

  • addCredit now uses Prisma increment (it previously decremented by mistake).
  • handlePayments-style paths moved toward atomic increments; webhook path is queue-driven.
  • Read-path tweaks: findUnique where appropriate, narrower selects, ordering on “my thumbnails”, combined creator fetch on detail route, explore without unused reference image payloads on list.
  • Waitlist uses upsert on unique email to avoid duplicate constraint noise.

04 Data model additions

GenerationJob — durable pipeline state: user, status string, prompts, image URL array, visibility, progress percent, optional result URL and thumbnail id, timestamps. Indexed by user + time and status + time.

WebhookEvent — idempotency anchor for Dodo: provider key + external webhook id unique, JSON payload, processing status, error message, processed timestamp.

Postgres remains authoritative for users, thumbnails, credits, and job audit rows. Redis holds ephemeral cache entries and short-lived progress blobs (with TTL); BullMQ stores queue metadata in Redis.

05 Local development notes

Run Redis locally (Docker map 6379:6379 or another host port). Use a TCP URL for BullMQ (redis://127.0.0.1:6379 or your mapped port). Upstash in production uses rediss:// when TLS is required — not the HTTP REST URL.

Set REDIS_URL in both web and worker environments. Worker loads dotenv via the npm script.

# Example REDIS_URL=redis://127.0.0.1:6379 # Processes (conceptually) pnpm dev # Next.js pnpm worker:dev # @repo/worker

Apply Prisma migrations so GenerationJob and WebhookEvent exist before enqueueing real traffic.
